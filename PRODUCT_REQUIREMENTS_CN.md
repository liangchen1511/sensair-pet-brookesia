# Sensair Pet 产品需求设计

> 文档版本：V0.2  
> 日期：2026-07-17  
> 当前硬件：ESP-SensairShuttle（ESP32-C5 N16R8、240 × 284 SPI LCD、触摸、麦克风、扬声器）

## 1. 产品定义

Sensair Pet 是一只常驻桌面的极简 AI 宠物，也是智能家居的自然语言入口。它通过黑底白眼动画表达状态，使用本地唤醒词进入小智语音会话，并通过 Home Assistant 控制家庭设备。

它不是“给智能音箱加一块屏幕”，也不是功能堆叠式开发板演示。产品的核心价值是：

1. 用户一眼就知道它正在等待、倾听、思考、回答还是出错。
2. 用户不用打开手机，就能完成一次自然的问答或家居控制。
3. 网络或云服务不可用时，它仍是一只有基本反应、可进入设置和重新配网的桌宠。

产品体验优先级固定为：

```text
声音连续可靠 > 唤醒与响应速度 > 触摸可控 > 表情状态正确 > 动画帧率 > 视觉复杂度
```

## 2. 目标用户与使用环境

### 2.1 目标用户

- 已使用 Home Assistant、米家或其他智能家居设备的个人用户。
- 希望桌面设备更有生命感，而不是只接受语音播报的用户。
- 乐于自行部署局域网服务、接受原型产品配置流程的开发者和数码爱好者。

### 2.2 典型环境

- 卧室、书房或办公桌，距离用户 0.5–2 米。
- USB 常供电，连接稳定的 2.4/5 GHz Wi-Fi。
- Home Assistant 运行在 NAS、树莓派、旧电脑或虚拟机上。

### 2.3 首版不服务的场景

- 远场多人会议、强噪声或音乐播放环境。
- 完全离线的大模型问答。
- 不依赖 Home Assistant、直接登录所有品牌云账号的一站式家居平台。
- 量产认证、儿童陪伴、医疗提醒及安全监控。

## 3. 产品原则

### 3.1 像一个角色，不像一块状态屏

- 主界面不常驻文字、按钮和菜单。
- 黑色背景与白色眼睛是默认视觉语言。
- 反馈优先用眼神、节奏和短提示音完成。
- 设置界面只在用户主动触摸进入时出现。

### 3.2 任何时候都不能“失控”

- 播报期间触摸屏幕必须能中断回答。
- 设置中必须提供重新配网、清除小智激活信息和恢复出厂设置。
- 云服务失败、HA 离线或资源包损坏时，设备仍可启动并给出明确反馈。

### 3.3 针对硬件能力设计体验

ESP32-C5 是单核芯片。首版明确采用音频优先的半双工模式：

- 用户说话时运行本地 AFE、唤醒和上传，降低动画负载。
- 小智播报时停止麦克风编码器和 AFE，把 CPU 让给网络收包、Opus 解码和音频播放。
- 播报期间不支持用唤醒词打断，使用触摸打断。
- 播报结束后自动恢复麦克风和本地唤醒。
- 显示来不及时允许丢弃过期动画帧，禁止为了补帧拖慢音频。

若未来产品必须支持自然的全双工语音打断，应迁移到双核 ESP32-S3/S31，而不是继续在 C5 上堆缓冲。

## 4. 核心用户旅程

### 4.1 首次使用

1. 设备启动并清黑屏，不显示残留白块。
2. 没有 Wi-Fi 凭据时自动进入 SoftAP 配网。
3. 用户用手机完成 Wi-Fi 配置。
4. 设备联网后初始化小智；如需激活，显示激活码并播放数字语音。
5. 激活完成后进入待机表情，并提示可以说“你好小智”。

### 4.2 日常对话

1. 待机状态播放低频眨眼动画。
2. 检测到“你好小智”后立即切换倾听表情和提示音。
3. 用户说话时保持倾听反馈，不显示长文本。
4. 识别结束后切换思考表情。
5. 收到 TTS 后切换说话表情并连续播放声音。
6. 回答结束后恢复本地唤醒并回到待机。

### 4.3 中断回答

1. 用户在播报期间点击屏幕。
2. 设备立即停止 TTS 和说话表情。
3. 麦克风/AFE 恢复后进入倾听状态。
4. 首版不承诺播报期间使用唤醒词打断。

### 4.4 重新配网

1. 从主界面滑动进入设置。
2. 点击“重新配网”。
3. 二次确认后仅清除 Wi-Fi 凭据并重启进入 SoftAP；默认保留亮度、音量、HA 和角色设置。
4. “恢复出厂设置”单独提供，并清除全部用户配置。

### 4.5 家居控制（P1）

1. 用户说“打开书房灯”。
2. 小智调用统一 HomeService 工具，而不是直接拼 HA 请求。
3. 宠物显示执行中表情。
4. 成功时给出简短语音和开心反馈；失败时说明目标、原因和建议。
5. 门锁、报警、加热等高风险动作必须语音或触摸二次确认。

## 5. 功能范围与优先级

### P0：稳定桌宠语音闭环

- SoftAP 首次配网、断线重连和重新配网。
- 小智激活、设备注册和会话连接。
- 本地“你好小智”唤醒。
- 倾听、思考、说话、待机、失败五类核心状态。
- 原生 Emote/EAF 黑白表情包，动画居中显示。
- 音频优先半双工调度和动画过期帧丢弃。
- 触摸中断播报、进入设置和恢复出厂设置。
- 无字幕主界面；设置、配网和激活页允许必要文字。
- 串口健康指标：CPU、内部内存、音频丢包、显示 DMA 和动画丢帧。

### P1：智能家居助手

- HA 地址、令牌和设备别名配置。
- 灯、开关、亮度、色温、空调温度、场景执行和状态查询。
- HomeService 统一接口与小智 MCP 工具注册。
- 操作成功、失败、超时和高风险确认反馈。
- HA 离线时不影响基础桌宠和语音问答。

### P2：环境感知与宠物人格

- BME690 环境数据；BMI270/BMM350 姿态和动作数据。
- 被触摸、被拿起、晃动、夜间和空气不舒适等自主反应。
- 可配置作息、情绪倾向和低频自主行为。
- 简单本地记忆，不在设备端保存敏感原始语音。

### P3：产品化

- OTA、故障回滚、隐私开关和日志导出。
- 统一外壳、声学腔体、麦克风距离和扬声器防啸叫设计。
- 评估 ESP32-S3/S31 双核硬件，实现全双工打断和更高帧率。
- 量产测试、认证和账号体系。

## 6. 表情与交互规范

| 状态 | 视觉反馈 | 音频策略 | 用户可执行操作 |
|---|---|---|---|
| 启动 | 黑屏后短暂睁眼 | 不播长音频 | 等待 |
| 配网 | 配网图标/二维码/必要文字 | 简短提示 | 手机配网、取消 |
| 待机 | 低频眨眼、轻微自主动作 | AFE 与唤醒开启 | 唤醒、点击、滑动 |
| 倾听 | 眼睛聚焦或轻微呼吸 | 录音和上传优先 | 说话、点击取消 |
| 思考 | 左右扫视或加载动作 | 网络优先 | 点击取消 |
| 说话 | 嘴型/眼睛节奏变化 | TTS 最高优先级，AFE 关闭 | 点击打断 |
| 成功 | 短暂弯眼笑 | 可选短提示音 | 无 |
| 失败 | 短暂低垂/警觉 | 一句可行动提示 | 重试、进设置 |
| 睡眠 | 闭眼、低亮度 | 保留低功耗唤醒能力 | 唤醒、点击 |

视觉约束：

- 资源画布必须为 240 × 284，中心与屏幕中心误差不超过 2 像素。
- 首帧显示前必须清空 LCD GRAM，禁止出现大面积白块残留。
- 主界面不显示对话字幕和语言文本。
- 待机目标 10–12 fps；语音高负载期间允许降至 6–8 fps并主动丢帧。
- 禁止补播过期帧；状态切换应优先于当前动画的完整播放。

## 7. 软硬件实现设计

### 7.1 当前 C5 原型

- ESP32-C5 N16R8，单核，Wi-Fi 6。
- ILI9341 SPI LCD，240 × 284，40 MHz，总线异步 DMA。
- CST816S 触摸。
- ADC 麦克风、I2S/Codec 扬声器输出。
- Brookesia 官方 SensairShuttle HAL。

推荐运行模式：

```text
待机：AFE + WakeNet + 低负载 Emote
倾听：AFE + Opus 编码/上传 + 降帧 Emote
思考：网络优先 + 低负载 Emote
说话：停止 AFE/编码器 + UDP/TTS 解码优先 + 允许丢动画帧
结束：重启 AFE/编码器 + 恢复待机
```

### 7.2 软件架构

- ESP-IDF 5.5.x：底层运行时。
- ESP-Brookesia chatbot：产品工程基线、服务生命周期和设置界面。
- `brookesia_agent_xiaozhi`：小智协议、激活和 Agent 状态。
- `brookesia_service_audio`：AFE、编码、解码和播放。
- `brookesia_expression_emote`：表情状态与原生动画资源。
- `AI_Agents`：Agent 事件到产品状态/表情的映射。
- `HomeService`（P1）：隔离 HA、未来米家直连和上层 MCP 工具。

禁止再创建平行的自制 XiaozhiBridge 或第二套 LCD flush 链路。上层扩展必须通过官方 Agent、Audio、Emote 和 HAL 接口完成。

### 7.3 后续硬件决策门槛

满足以下任一条件时，进入 S3/S31 硬件评估：

- 产品必须支持语音打断 TTS。
- C5 在停掉 AFE 后仍无法连续播放 TTS。
- 希望对话期间稳定达到 12–15 fps。
- 同时启用传感器融合、复杂动画和本地语音处理后 CPU 长期超过 80%。

## 8. 非功能需求与验收指标

### 8.1 音频

- 安静室内、1 米距离，本地唤醒成功率目标 ≥ 90%。
- 连续完成 10 轮对话，不出现可感知的周期性断音或爆音。
- TTS 期间 `Received audio packet with wrong sequence` 应降至偶发；产品目标为单轮会话 0 次。
- 点击中断到声音停止的时间 ≤ 300 ms。
- TTS 结束后 1.5 秒内恢复本地唤醒能力。

### 8.2 显示

- 上电后 200 ms 内完成全屏清黑，不出现持续白块。
- 表情中心误差 ≤ 2 像素，无右偏、裁切和残影。
- 显示异步 DMA `submit == done`，持续运行无 flush 超时和 watchdog。
- 动画丢帧不能造成状态倒退、补帧加速或影响音频。

### 8.3 稳定性与资源

- 连续运行 8 小时无崩溃、重启、任务看门狗或明显内存增长；P1 提升到 24 小时。
- 对话峰值总 CPU 应从当前约 91–92% 降到 80% 以下；TTS 阶段应留出可测的 Idle 时间。
- 内部可用内存最低值建议 ≥ 16 KB，最大连续块 ≥ 8 KB。
- 资源包损坏或缺失时进入安全 fallback，不阻塞配网和设置。

### 8.4 网络与配置

- 保存过的 Wi-Fi 在正常信号下 30 秒内自动重连。
- 重新配网与恢复出厂设置必须是两个独立动作，并有二次确认。
- HA Token、小智凭据和 Wi-Fi 密码不得硬编码进仓库或打印到日志。

## 9. 当前实现评估

### 已达到

- ESP-SensairShuttle 官方 HAL、LCD、触摸和音频硬件可正常启动。
- SoftAP 配网、重新配网、小智激活、本地唤醒和云端对话链路已经跑通。
- 原生黑白 Emote 资源已接入，文字主界面已移除。
- 显示已使用异步 DMA，提交与完成计数稳定，过期动画帧可主动丢弃。
- 上电白块已通过首屏清黑处理，仍需以本轮实机结果最终确认。

### 当前阻塞

- 日志显示对话峰值 CPU 约 91–92%，其中麦克风录制/AFE 约占 52–53%，音频 feeder 约占 20%。
- Wi-Fi RSSI 约 −34 dBm，说明卡顿主要不是弱网，而是单核 CPU 饱和。
- TTS 中持续出现音频包序号错误，动画每 5 秒丢弃约 36–52 帧，峰值更高。
- 内部内存通常只剩约 18–21 KB，继续盲目增加网络缓冲会扩大风险。

### 本阶段决策

进入第二阶段，但只做性能稳定化，不开始 Home Assistant 功能：

1. C5 半双工播报时完全停止麦克风编码器和 AFE。
2. 播报结束后自动恢复本地唤醒。
3. 继续保持音频优先和动画过期帧丢弃。
4. 实机验证 CPU、序号错误、声音连续性和唤醒恢复。
5. 只有通过性能门槛后，才进入 P1 Home Assistant。

## 10. 里程碑

### M0：稳定语音桌宠（当前）

- M0.1 显示链路稳定、无白块、动画居中。
- M0.2 音频优先半双工，TTS 无明显卡顿。
- M0.3 触摸中断、重新配网和异常恢复通过。
- Gate：10 轮连续对话 + 8 小时稳定运行通过。

### M1：Home Assistant 闭环

- 配置 HA 地址、令牌和别名。
- 一盏灯、一个插座/空调、一个场景完成语音与触摸控制。
- 高风险动作二次确认。
- Gate：HA 离线不影响桌宠基础模式。

### M2：环境桌宠

- 接入环境与姿态传感器。
- 建立自主行为、睡眠和环境提醒。
- Gate：传感器缺失可降级，连续运行无资源回退。

### M3：产品化验证

- 比较 C5 与 S3/S31 体验、BOM 和功耗。
- 完成声学、外壳、OTA、隐私和故障恢复设计。
- 决定原型版与产品版硬件基线。

## 11. 暂不做与待确认事项

暂不做：完整养成数值、商城、账号社交、摄像头视觉、复杂全身角色动画、米家非公开接口直连。

待后续验证后再决策：

- C5 半双工优化能否满足声音连续性门槛。
- 用户是否能接受“播报期间触摸打断、不能唤醒词打断”。
- Home Assistant 配置应放在设备设置页还是手机配网页。
- 产品版是否直接采用 S3/S31，C5 仅保留为开发板原型。
