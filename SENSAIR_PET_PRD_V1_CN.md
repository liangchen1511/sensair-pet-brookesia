| 项目 | 内容 |
|---|---|
| 产品名称 | Sensair Pet 桌面共生宠物 |
| 文档版本 | V1.0 |
| 文档状态 | 原型开发基线 |
| 更新时间 | 2026-07-17 |
| 首发硬件 | ESP-SensairShuttle / ESP32-C5 N16R8 |
| 软件基线 | ESP-IDF 5.5.4 + ESP-Brookesia chatbot + XiaoZhi Agent + Emote/EAF |
| 产品阶段 | P0 语音与表情体验稳定化 |

## 1. 一句话定义

Sensair Pet 是一只常驻桌面的黑白眼睛 AI 宠物：它能用表情让用户看懂自己的状态，通过“你好小智”完成语音对话，并在后续版本中成为 Home Assistant 的自然语言入口。

它首先是一只有明确反馈、不会失控的小生命，其次才是智能音箱、状态屏和家居遥控器。

## 2. 产品背景与机会

当前语音助手常见的体验问题是：

- 用户不知道设备有没有听见、正在思考还是已经失败。
- 语音助手与桌面环境缺少长期情感连接，用完即走。
- 智能家居入口分散在多个 App，控制路径长。
- 开发板演示往往功能很多，但声音、动画、异常恢复和重新配网不够可靠。

ESP-SensairShuttle 已具备屏幕、触摸、麦克风、扬声器、Wi-Fi 和可扩展传感器，适合做一个有状态表达能力的桌面终端。Sensair Pet 的机会不是与大型智能音箱比音质和远场能力，而是用近场桌面场景、极简表情和可扩展家居能力，建立更轻、更有生命感的交互。

## 3. 产品目标

### 3.1 P0 目标

完成一个可以每天放在桌上使用的语音桌宠闭环：

1. 能可靠启动、配网、重新配网和自动重连。
2. 能通过本地唤醒词进入小智对话。
3. 声音连续，动画不会反过来拖慢音频。
4. 用户始终能从表情判断设备状态。
5. 用户可以通过触摸打断播报、进入设置和恢复配置。
6. 连续运行不崩溃、不白屏、不触发看门狗。

### 3.2 P1 目标

成为 Home Assistant 的桌面入口，覆盖高频灯光、插座、空调和场景控制，并提供执行结果与异常反馈。

### 3.3 商业与传播目标

- 形成一个可演示、可复刻、可继续换角色素材的开源原型。
- 通过“黑镜白眼 + 状态动画”建立清晰的产品辨识度。
- 用 C5 原型验证真实需求，再决定是否投入双核产品硬件。

## 4. 非目标

首版不做：

- 设备端大模型或完全离线问答。
- 远场多人拾音和高保真音乐播放。
- 摄像头、视觉识别和视频通话。
- 完整数值养成、商城、社交和账号体系。
- 绕过 Home Assistant 直接适配全部米家私有云接口。
- 依赖语音打断 TTS 的全双工体验。
- 电池供电和全天移动使用。
- 面向儿童、医疗、安全报警等高责任场景。

## 5. 目标用户

### 5.1 核心用户

智能家居爱好者、开发者和数码玩家：

- 已有 Home Assistant、米家、NAS 或其他常开设备。
- 愿意完成一次配网和 Token 配置。
- 希望桌面设备比传统智能音箱更有生命感。
- 接受首版为可靠原型，而不是免配置量产消费品。

### 5.2 次级用户

- 希望在书房或办公室使用轻量 AI 问答的用户。
- 希望换角色、换表情包或二次开发 MCP 工具的创作者。

### 5.3 使用环境

- 距离用户约 0.5–2 米的安静室内桌面。
- 5 V USB 常供电。
- Wi-Fi 信号稳定，通常优于 −65 dBm。
- 设备屏幕正对或略低于用户视线。

## 6. 开源基线与复用策略

本产品不重新发明已有的设备框架，优先组合经过适配的开源模块：

| 能力 | 复用基线 | 产品侧工作 |
|---|---|---|
| 产品工程 | `esp-brookesia/examples/agent/chatbot` | 删除无关演示能力，加入产品状态、性能策略和设置项 |
| 小智对话 | `brookesia_agent_xiaozhi` | 保留官方激活、连接、状态事件与音频协议 |
| 板级驱动 | `brookesia_hal_boards/esp_sensair_shuttle` | 固定屏幕方向、SPI 参数、音频通道和触摸映射 |
| 音频服务 | `brookesia_service_audio` + `av_processor` | C5 半双工资源调度、性能观测和异常恢复 |
| 本地唤醒 | ESP-SR AFE / WakeNet | 使用“你好小智”，按产品状态启停 |
| 表情播放 | `brookesia_expression_emote` + `esp_emote_gfx` + EAF | 黑底白眼资源、状态映射、居中与丢帧策略 |
| MCP 工具 | Brookesia / `esp-agents-firmware` 的工具注册模式 | 注册设备设置与 Home Assistant 工具 |
| 配网设置 | Brookesia Wi-Fi Service + chatbot 设置页 | 重新配网、恢复出厂设置和配置状态反馈 |

架构原则：

- 不再创建第二套 XiaoZhiBridge。
- 不允许 LVGL 和 Emote 各自注册一套 LCD 完成回调。
- Agent、Audio、Emote 和 HAL 只保留一条主链路。
- 产品逻辑通过官方事件和服务接口扩展，避免长期维护私有协议分支。

## 7. 硬件性能边界

### 7.1 当前硬件

| 项目 | 当前配置 | 对产品的影响 |
|---|---|---|
| SoC | ESP32-C5，单核 RISC-V，240 MHz | 不能同时追求重 AFE、全双工语音、网络解码和高帧率动画 |
| Flash | 16 MB | 足够容纳双应用/资源分区，但需控制字体、多 Agent 和重复资源 |
| PSRAM | 8 MB，80 MHz | 适合动画、任务栈和大缓冲；不能替代 DMA/实时音频所需内部 SRAM |
| 内部 SRAM | 日志峰值仅剩约 18–21 KB，最大连续块约 14 KB | 禁止继续盲目增加 UDP、DMA 和解码缓冲 |
| 显示 | ILI9341，240×284，RGB565，SPI 40 MHz | 适合黑白局部动画；不适合持续全屏 25 fps 刷新 |
| 触摸 | CST816S，I2C | 用于点击打断、滑动进入设置和确认危险操作 |
| 麦克风 | 单声道，16 bit，16 kHz，模拟 ADC 路径 | 定位近场语音，不承诺阵列远场和复杂波束成形 |
| 扬声器 | I2S/Codec 输出 | 以清晰 TTS 为目标，不以音乐音质为目标 |
| 网络 | Wi-Fi 6；现有日志 RSSI 约 −34 dBm | 当前卡顿主要是 CPU 饱和，不是弱网 |

### 7.2 当前实测负载

旧版本对话期间：

- 总 CPU 使用约 91–92%。
- `audio_recorder_` / AFE 约占 52–53%。
- `audio_feeder_ta` 约占 20%。
- 内部可用内存约 18–21 KB。
- TTS 中多次出现 UDP 音频包序号错误。
- 动画每 5 秒丢弃约 36–52 个过期帧，峰值更高。

因此 C5 版本采用以下硬约束：

```text
待机：开启 AFE/WakeNet，表情低负载运行
倾听：AFE + Opus 上传优先，动画主动降帧
思考：网络优先，表情使用轻量循环
说话：停止 recorder/AFE/encoder，优先 UDP 收包、解码和播放
结束：恢复 recorder/AFE/encoder，再返回待机
```

### 7.3 C5 体验承诺

- 采用半双工，不承诺 TTS 期间语音唤醒打断。
- TTS 期间点击屏幕可以打断。
- 动画时间线可以跳帧，但不能补播过期帧。
- 待机动画目标 10–12 fps；语音高负载期间允许 6–8 fps。
- 资源优先级固定为：音频 > 网络 > 触摸 > 状态正确 > 动画帧率。

### 7.4 产品版芯片决策

出现以下任一需求时，启动 ESP32-S3/S31 双核产品路线：

- 必须支持播报期间语音打断。
- 对话期间要求稳定 12–15 fps。
- 同时运行 AFE、多个传感器融合和复杂动画。
- C5 停止 AFE 后，TTS 仍存在明显卡顿或包序号错误。
- 需要更完整的本地语音或视觉处理。

C5 定位为成本可控的原型/开发者版本；双核版本定位为更接近消费产品的体验版本。

## 8. 产品形态设计

### 8.1 外观概念

推荐采用“圆角黑镜桌宠”形态：

- 正面只保留纵向屏幕，息屏时形成完整黑色镜面。
- 白色眼睛位于视觉中心，不常驻文字和图标。
- 麦克风开孔位于顶部或正面上沿，避开扬声器声路。
- 扬声器向后或向下出声，通过桌面反射扩大近场覆盖。
- USB、电源和调试口放在背面，正面不暴露开发板结构。
- 底部加防滑垫并保持轻微后仰，避免屏幕反光和触摸倾倒。
- 传感器子板设计为背部/底部可拆模块，不让 P0 被全套传感器绑定。

### 8.2 物理交互

- 单击：开始对话；播报时中断并进入倾听。
- 长按：取消当前会话或返回待机。
- 左右/边缘滑动：进入或退出设置。
- 设置页危险操作：必须点击确认，避免误清配置。
- 无实体静音键的原型版，设置中必须提供麦克风隐私开关；产品版建议增加物理静音键和明显指示。

## 9. 核心体验状态机

```text
启动
 ├─ 无 Wi-Fi → 配网 → 联网
 └─ 有 Wi-Fi → 自动联网
联网 → 小智激活/连接 → 待机
待机 → 唤醒/点击 → 倾听 → 思考 → 说话 → 待机
说话 → 点击打断 → 倾听
任意状态 → 网络/服务失败 → 失败反馈 → 重试或待机
任意主界面 → 滑动 → 设置 → 返回主界面
```

状态切换必须由 Agent/Service 事件驱动，不能依赖固定延时猜测云端状态。

## 10. 功能需求

### FR-01 启动与自检（P0）

要求：

- 上电先清黑 LCD GRAM，再加载 UI 和动画。
- 初始化 Flash、PSRAM、LCD、触摸、音频、Wi-Fi 和资源分区。
- 非关键模块失败时进入降级模式，不允许无限重启。
- 串口输出硬件结果和可行动错误原因。

验收：

- 上电后 200 ms 内完成全屏清黑，不出现持续白块。
- 30 秒内进入配网、联网或明确错误状态。
- 连续冷启动 20 次无白屏和任务看门狗。

### FR-02 Wi-Fi 配网与重配（P0）

要求：

- 无凭据时自动进入 SoftAP 配网。
- 已保存凭据时自动连接并重试。
- 设置页提供“重新配网”和“恢复出厂设置”两个独立动作。
- 重新配网只清 Wi-Fi；恢复出厂设置清全部用户配置。

验收：

- 正常信号下保存的网络 30 秒内连通。
- 重新配网后自动进入 SoftAP。
- 错误密码、路由器离线和网络切换均有可理解反馈。

### FR-03 小智激活与连接（P0）

要求：

- 复用官方 XiaoZhi Agent 激活流程。
- 需要激活时显示激活码，并播放数字语音。
- 激活信息安全保存，重启后无需重复激活。
- 服务连接失败时指数退避重试，不阻塞设置界面。

验收：

- 新设备可独立完成激活。
- 已激活设备重启后自动恢复会话能力。

### FR-04 本地唤醒与倾听（P0）

要求：

- 默认唤醒词为“你好小智”。
- 待机时开启 AFE/WakeNet。
- 唤醒成功后 300 ms 内切到倾听表情并播放短提示音。
- 用户说话期间音频上传优先，动画允许降帧。

验收：

- 安静室内 1 米距离，20 次测试唤醒成功率不低于 90%。
- 不说唤醒词的 8 小时待机中，误唤醒次数目标不高于 2 次。

### FR-05 语音回答（P0）

要求：

- 收到 TTS 后进入说话状态。
- C5 说话状态停止 recorder、AFE 和 encoder。
- 音频播放拥有高于动画渲染的任务优先级。
- 播报结束后自动恢复 AFE 和本地唤醒。

验收：

- 连续 10 轮对话无明显周期性断音、爆音和重启。
- 单轮会话目标为 0 次 `wrong sequence`；阶段门槛为较旧版本显著下降且无可感知断音。
- 播报结束 1.5 秒内重新具备本地唤醒能力。

### FR-06 触摸打断（P0）

要求：

- 说话时单击屏幕发送官方 InterruptSpeaking。
- 立即停止音频并进入倾听。
- 其他状态单击可手动开始对话或取消当前操作。

验收：

- 从触摸到声音停止不超过 300 ms。
- 连续打断 20 次无状态卡死。

### FR-07 表情系统（P0）

要求：

- 黑色背景、白色眼睛，主界面不显示字幕。
- 至少覆盖待机、倾听、思考、说话、成功、失败、睡眠。
- 动画资源使用原生 Emote/EAF 结构，不经第二套 LCD flush。
- 表情中心与屏幕中心误差不超过 2 像素。
- 动画超时时跳至正确时间线，不补播旧帧。

验收：

- 表情无右偏、裁切、残影和上电白块。
- 显示 DMA 的提交数与完成数一致，无 flush 超时。
- 动画丢帧不造成音频卡顿和状态倒退。

### FR-08 设置与隐私（P0）

设置项：

- Wi-Fi 状态与重新配网。
- 音量和屏幕亮度。
- 麦克风软件静音。
- 小智连接/激活状态。
- 设备信息、版本和恢复出厂设置。

约束：主表情页无文字不等于整个产品无文字；配网、激活、设置和错误页允许必要文字。

### FR-09 Home Assistant 控制（P1）

要求：

- 通过统一 `HomeService` 隔离上层和 HA REST/WebSocket 结构。
- 支持灯、开关、亮度、色温、空调温度、场景和状态查询。
- 用户别名映射到 HA `entity_id`。
- 通过 XiaoZhi MCP 工具调用，不让模型直接拼 HTTP 请求。
- 门锁、报警、加热等高风险动作要求二次确认。

验收：

- 至少完成一盏灯、一个插座/空调和一个场景的真实控制。
- HA 离线时桌宠和普通问答仍可使用。
- Token 不硬编码、不打印到日志。

### FR-10 环境与姿态反应（P2）

要求：

- BME690 提供环境快照；BMI270/BMM350 提供姿态和动作上下文。
- 支持被拿起、轻晃、环境不舒适、夜间调暗等低频自主行为。
- 传感器未插入时自动降级，不影响 P0。

## 11. 表情与反馈规范

| 产品状态 | 表情 | 声音 | 资源策略 |
|---|---|---|---|
| 启动 | 黑屏后睁眼 | 无或极短启动音 | 先清屏，禁止加载大字体 |
| 配网 | 配网符号/必要文字 | 一次简短提示 | 停止高频动画 |
| 待机 | 眨眼、轻微扫视 | AFE 工作，无持续声音 | 10–12 fps，低频自主动作 |
| 倾听 | 聚焦、轻呼吸 | 录音/上传最高优先 | 6–8 fps，可跳帧 |
| 思考 | 左右扫视/点状节奏 | 网络优先 | 使用短循环，避免全屏刷新 |
| 说话 | 眼睛/嘴型节奏 | TTS 最高优先 | 6–8 fps，停止 AFE，可跳帧 |
| 成功 | 弯眼笑 | 可选短提示 | 一次性动画后回待机 |
| 失败 | 低垂或警觉 | 一句可行动提示 | 禁止无限循环报警 |
| 睡眠 | 闭眼、低亮度 | 保留唤醒 | 低刷新或静态画面 |

## 12. 非功能需求

### NFR-01 稳定性

- 开发验收：连续运行 8 小时无崩溃、重启、看门狗和明显内存增长。
- P1 验收：连续运行 24 小时。
- 服务失败不能导致配网和设置失效。

### NFR-02 性能

- TTS 阶段总 CPU 目标低于 80%，必须出现可测 Idle 时间。
- 内部可用内存最低建议不低于 16 KB，最大连续块不低于 8 KB。
- 显示 DMA `submit == done`，错误计数为 0。
- 禁止通过持续扩大缓冲掩盖 CPU 饱和。

### NFR-03 安全与隐私

- Wi-Fi 密码、HA Token 和云端凭据保存于 NVS，不进入仓库和普通日志。
- 恢复出厂设置必须清除用户凭据。
- 设置页清楚显示麦克风是否启用。
- 产品版增加物理麦克风静音能力。

### NFR-04 可维护性

- 上游组件修改必须有独立 Kconfig 开关和迁移记录。
- 产品状态与动画资源名通过映射配置关联。
- Home Assistant、未来米家直连和 Agent 不得互相直接依赖。
- 串口性能日志应可通过发布配置关闭。

## 13. 异常处理矩阵

| 异常 | 用户反馈 | 自动处理 | 不允许发生 |
|---|---|---|---|
| Wi-Fi 断开 | 断网表情/设置状态 | 后台重连 | 无限阻塞主界面 |
| 小智不可用 | 一句失败提示 | 退避重连 | 连续播放错误音 |
| 激活失败 | 显示激活状态 | 允许重试/清除激活 | 打印敏感凭据 |
| HA 离线 | 家居操作失败表情 | 不影响普通问答 | 整机重启 |
| 动画资源损坏 | fallback 表情 | 保留设置与配网 | 黑屏卡死 |
| 传感器缺失 | 设置页显示不可用 | 跳过传感器服务 | 阻止 P0 启动 |
| 内存不足 | 降低动画/拒绝非核心任务 | 保留音频和设置 | 继续盲目分配直至崩溃 |
| 音频服务异常 | 停止会话并恢复待机 | 重启音频链路 | 播放随机噪声 |

## 14. 版本规划

### M0：稳定语音桌宠

- 显示无白块、动画居中。
- 配网、重配、激活和唤醒完整。
- C5 音频优先半双工通过。
- 触摸打断和异常恢复通过。

发布门槛：10 轮连续对话、20 次冷启动、8 小时稳定运行。

### M1：Home Assistant MVP

- 完成配置、别名、查询和控制。
- 灯、插座/空调、场景各一个真实设备通过。
- 高风险操作确认和 HA 离线降级通过。

### M2：环境桌宠

- 传感器快照、动作反应和夜间模式。
- 低频自主行为和基础人格配置。

### M3：产品化验证

- 对比 C5 与 S3/S31 的体验、BOM、功耗和软件复杂度。
- 完成外壳、声学腔体、物理静音、OTA 和回滚。
- 决定开发者版与产品版硬件分型。

## 15. 核心指标

| 指标 | P0 门槛 | 产品目标 |
|---|---:|---:|
| 1 米安静环境唤醒成功率 | ≥90% | ≥95% |
| 触摸中断延迟 | ≤300 ms | ≤200 ms |
| TTS 结束后恢复唤醒 | ≤1.5 s | ≤800 ms |
| 连续对话 | 10 轮无明显断音 | 30 轮无异常 |
| 冷启动 | 20 次无白屏/重启 | 100 次 |
| 稳定运行 | 8 小时 | 7×24 小时 |
| 表情中心偏差 | ≤2 px | ≤1 px |
| TTS 阶段 CPU | <80% | <70% |
| 音频包序号错误 | 显著下降且无可感知断音 | 单轮 0 次 |

## 16. 当前产品决策

已经确定：

1. P0 只使用 XiaoZhi，不把多 Agent 切换作为用户卖点。
2. 主界面采用无字幕黑底白眼；必要文字只存在于功能页。
3. C5 使用音频优先半双工，播报期间触摸打断。
4. Home Assistant 是 P1，不阻塞语音桌宠稳定化。
5. 传感器是增强模块，不阻塞基础产品启动。
6. 动画掉帧可以接受，声音断续不可接受。
7. 若 C5 无法跨过音频稳定门槛，停止微调并迁移双核硬件。

## 17. 待确认事项

- “MDK”若指外壳机械设计套件，需要补充板框尺寸、接口位置和可复用 3D 模型来源。
- 产品版是否保留纵向 240×284 屏，或改用更规则的方形屏幕。
- Home Assistant 配置放在设备设置页、SoftAP 网页还是配套手机页面。
- 是否接受 C5 版播报期间只能触摸打断。
- 产品是否需要物理麦克风静音键和独立隐私指示灯。
- 传感器子板是标配还是可选配件。
